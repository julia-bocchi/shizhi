# 前后端联调接口文档

> 基于当前仓库代码与 `mock-backend/server.js` 重新整理，适用于 2026-04-22。
> 目标：让 Harmony 前端和本地 / 真实后端按当前实现直接联调。
> 说明：本文尽量保持现有请求体、响应体字段不变，只修正文档与代码不一致的部分，并补充登录接口。

## 1. 联调范围

当前前端实际接入后端的模块有：

1. 登录
2. 用户资料
3. 体重记录
4. 食物类型与饮食计划
5. 训练类型、模板、训练计划、训练记录

当前 AI 助手与推荐能力默认直接请求第三方大模型接口，不走这套业务后端；`mock-backend` 中 `POST /api/v1/ai-assistant/chat` 目前返回 `501`，因此本文不再把它作为必接联调接口。

## 2. 通用约定

### 2.1 Base URL

默认联调地址：

`http://127.0.0.1:3000/api/v1`

前端常量位置：

`entry/src/main/ets/models/AuthSession.ets`

### 2.2 统一响应包裹

前端通过 `ApiClient.requestApiData` 解析固定结构：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

说明：

- `code = 0` 表示成功。
- HTTP 状态码 `>= 400` 时，前端会直接按失败处理。
- 即使 HTTP 200，只要 `code != 0`，前端也会按失败处理。

### 2.3 鉴权方式

登录成功后，前端后续请求会自动携带：

```http
Authorization: Bearer <token>
X-User-Id: <userId>
Content-Type: application/json
```

说明：

- 前端是否视为已登录，只看 `userId` 和 `token` 是否非空。
- mock 后端当前支持从 `Authorization` / `X-User-Id` / query 中识别用户。

### 2.4 通用字段规则

| 字段 | 说明 |
| --- | --- |
| `date` | 固定 `YYYY-MM-DD` |
| `id` | 当前前端统一按 `string` 使用 |
| `weight` | 单位 `kg`，支持小数 |
| `calories` | 单位 `kcal`，支持小数 |
| `durationSeconds` | 单位秒 |
| `accentColor` | 十六进制颜色值 |

### 2.5 错误码建议

| code | 含义 |
| --- | --- |
| `0` | 成功 |
| `400` | 参数错误 |
| `401` | 未登录或 token 失效 |
| `404` | 数据不存在 |
| `422` | 业务校验失败 |
| `500` | 服务异常 |

## 4. 登录

### 4.1 账号密码登录

`POST /api/v1/auth/login`

请求体：

```json
{
  "username": "demo",
  "password": "123456"
}
```

响应体：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "userId": "user-demo",
    "username": "demo",
    "token": "mock-token-user-demo"
  }
}
```

说明：

- 前端登录页只要求 `username`、`password` 两个字段。
- 当前 mock 后端只校验非空，不校验固定密码；不存在的账号会自动创建默认用户。
- 登录成功后，前端会立刻继续请求 `GET /user-profile` 拉取用户资料。

## 6. 体重记录

### 6.1 获取体重列表

`GET /api/v1/weight-records?startDate=2026-04-01&endDate=2026-04-22`

响应体：

```json
{
  "code": 0,
  "message": "ok",
  "data": [
    {
      "id": "weight-1",
      "date": "2026-04-15",
      "weight": 58.9
    },
    {
      "id": "weight-2",
      "date": "2026-04-16",
      "weight": 58.7
    }
  ]
}
```

说明：

- 这里的 `data` 是数组，不是 `{ items: [] }`。
- 前端会按列表直接渲染趋势、最近记录和月视图。

### 6.2 获取体重汇总

`GET /api/v1/weight-records/summary?days=7`

响应体：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "latestRecord": {
      "id": "weight-5",
      "date": "2026-04-19",
      "weight": 58.2
    },
    "previousRecord": {
      "id": "weight-4",
      "date": "2026-04-18",
      "weight": 58.4
    },
    "recentCount": 5,
    "averageWeight": 58.54,
    "changeInRange": -0.7,
    "changeFromPrevious": -0.2
  }
}
```

### 6.3 按日期新增或覆盖体重

`PUT /api/v1/weight-records/{date}`

示例：

`PUT /api/v1/weight-records/2026-04-22`

请求体：

```json
{
  "weight": 58.1
}
```

响应体：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "id": "weight-6",
    "date": "2026-04-22",
    "weight": 58.1,
    "isUpdated": false
  }
}
```

说明：

- 建议唯一键：`userId + date`。
- `isUpdated = false` 表示新增，`true` 表示覆盖已有记录。

## 12. 与当前代码强相关的补充说明

### 12.1 当前不同接口的 `data` 形态并不完全一致

这是当前前端真实兼容情况，后端联调时请保持：

| 接口 | `data` 形态 |

### 12.2 离线同步不是单独接口

当前 `SyncQueue` / `SyncService` 的做法是：

- 本地先保存
- 再把变更排队
- 最终重放到上面这些业务接口

也就是说后端不用额外提供“批量同步接口”，只要保证上述业务接口幂等或可重复调用即可。

### 12.3 建议优先保证的最小联调链路

1. `POST /auth/login`
2. `GET /user-profile` + `PUT /user-profile`
3. `GET /weight-records` + `PUT /weight-records/{date}`
4. `GET /workout-types` + `GET /workout-plans` + `POST /workout-plans`
5. `GET /workout-records` + `POST /workout-records` + `GET /workout-records/daily-summary`
