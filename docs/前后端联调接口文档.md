# 前后端联调接口文档

> 基于当前仓库页面与模型整理，适用于 2026-04-17 版本。
> 覆盖范围：用户页、食物页、AI 助手页、体重页、训练页。
> 说明：当前食物页模型和存储键已存在，但还没有挂到首页 Tab；本文按现有数据结构先给出可直接落地的接口定义。

## 1. 文档目标

这份文档的目标不是做一套很重的后端设计，而是让前端现在这几个页面可以顺利联调：

1. 用户页：读写用户资料。
2. 食物页：读取食物类型、维护饮食计划。
3. AI 助手页：发送问题、拿到建议和训练计划草案。
4. 体重页：记录当天体重、读取趋势数据。
5. 训练页：先读取用户可用的运动类型，再读取已有计划并渲染日历和当日计划。

## 3. 通用约定

### 3.1 Base URL

`/api/v1`

### 3.2 统一返回结构

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

### 3.3 通用字段规则

| 字段 | 规则 |
| --- | --- |
| `date` | 固定使用 `YYYY-MM-DD` |
| `id` | 建议统一由后端生成，类型可为 string |
| `weight` | 单位 `kg`，支持小数 |
| `calories` | 单位 `kcal`，支持小数 |
| `durationSeconds` | 单位秒 |
| `accentColor` | 前端直接使用十六进制颜色值 |

### 3.4 登录态约定

- 正式环境默认后端从 token 识别用户。
- 联调阶段如果暂时没有登录态，可以临时用 header 或 query 传 `userId`。

### 3.5 错误码建议

| code | 含义 |
| --- | --- |
| `0` | 成功 |
| `400` | 参数错误 |
| `401` | 未登录或 token 失效 |
| `404` | 数据不存在 |
| `409` | 唯一键冲突 |
| `422` | 业务校验失败 |
| `500` | 服务异常 |

## 5. 用户页

### 5.1 获取用户资料

`GET /api/v1/user-profile`

响应体：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "nickname": "小林",
    "heightCm": "170",
    "weightKg": "68.5",
    "age": "26",
    "gender": "男"
  }
}
```

### 5.2 保存用户资料

`PUT /api/v1/user-profile`

请求体：

```json
{
  "nickname": "小林",
  "heightCm": "170",
  "weightKg": "68.5",
  "age": "26",
  "gender": "男"
}
```

说明：

- 当前前端全部用字符串保存，后端首版建议直接兼容字符串，联调最省事。
- `nickname` 可空。
- `weightKg` 对训练热量估算最关键。
- `heightCm + weightKg + age + gender` 会影响 BMI 和训练建议文案。

## 7. AI 助手页

### 7.1 页面现状

AI 助手页当前有 3 类行为：

1. 自由对话 `chat`
2. 趋势分析 `analyze_weight`
3. 生成训练计划草案 `generate_plan`

前端本地已经具备以下上下文：

- 用户资料 `UserProfile`
- 最近体重记录 `WeightRecord[]`
- 最近训练记录 `WorkoutRecord[]`
- 当前已有计划数量
- 已保存自定义训练模板名称
- 历史对话消息

### 7.2 AI 对话接口

`POST /api/v1/ai-assistant/chat`

请求体建议：

```json
{
  "task": "generate_plan",
  "message": "请结合我最近的训练和体重变化，为我生成接下来几天的训练计划。",
  "historyMessages": [
    {
      "role": "assistant",
      "content": "我是你的 AI 训练助手。"
    },
    {
      "role": "user",
      "content": "我最近体重有点反弹。"
    }
  ],
  "contextSnapshot": {
    "profile": {
      "nickname": "小林",
      "heightCm": "170",
      "weightKg": "68.5",
      "age": "26",
      "gender": "男"
    },
    "weightRecords": [
      {
        "id": 11,
        "date": "2026-04-16",
        "weight": 68.9
      },
      {
        "id": 12,
        "date": "2026-04-17",
        "weight": 68.5
      }
    ],
    "workoutRecords": [
      {
        "id": "record-1",
        "date": "2026-04-16",
        "name": "跑步",
        "calories": 286.4,
        "durationSeconds": 1800
      }
    ],
    "existingPlanCount": 2,
    "savedWorkoutNames": [
      "下班后跳绳"
    ]
  }
}
```

说明：

- 如果后端已能自己读取用户资料、体重、训练记录，那么 `contextSnapshot` 可以不传。
- 但联调首版建议保留，方便直接对齐当前前端实现。

响应体建议直接返回前端可用结构：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "reply": "你最近体重有回落，建议继续保持每周 3 到 4 次运动，并把高强度和恢复日穿插开。",
    "plans": [
      {
        "id": "draft-1",
        "date": "2026-04-18",
        "title": "晨间跑步",
        "summary": "配速 + 路程 · 30 分钟 · 286 kcal",
        "notes": "以恢复节奏开始",
        "draftText": "running%7C%7Cduration%7Cmoderate%7C30%7C5%7C6%7C18%7C300%7C120%7C4%7C40%7C20",
        "accentColor": "#0F6BFF"
      }
    ]
  }
}
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `task` | `chat`、`analyze_weight`、`generate_plan` |
| `reply` | 给用户展示的回复文案 |
| `plans` | 仅在生成计划时返回，其他场景返回空数组 |
| `draftText` | 训练草稿序列化结果，训练页导入时最省改动 |

### 7.3 AI 计划导入方式

AI 助手页不需要单独设计“导入计划”接口，直接复用训练计划新增接口：

`POST /api/v1/workout-plans`

也就是说：

1. AI 接口返回草案。
2. 用户确认后，前端把草案转成训练计划。
3. 再调用训练计划新增接口持久化。

## 9. 训练页

### 9.1 推荐加载时序

按你现在的想法，训练页建议采用下面这条链路：

1. 先请求当前用户有哪些运动类型：`GET /api/v1/workout-types`
2. 再请求当前月份或当前视图范围内的已有计划：`GET /api/v1/workout-plans`
3. 今日累计卡片可并行请求：`GET /api/v1/workout-records/daily-summary?date=YYYY-MM-DD`

这样做的好处：

- 页面一进来就能先把“预设训练 + 已保存模板”渲染出来。
- 计划接口只负责日历和当日计划，不和运动类型混在一起。
- 后续训练页扩展筛选、模板分组时，结构会更稳定。

### 9.2 获取运动类型

`GET /api/v1/workout-types`

说明：

- 这个接口建议合并返回两类数据：
  - 预设训练类型
  - 当前用户自定义模板
- 前端训练页可以直接按 `source` 分区展示。

响应体：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "items": [
      {
        "id": "running",
        "source": "preset",
        "exerciseId": "running",
        "savedWorkoutId": "",
        "name": "跑步",
        "description": "根据配速和路程估算总时长与热量。",
        "mode": "distance_pace",
        "intensity": "",
        "accentColor": "#0F6BFF"
      },
      {
        "id": "template-1001",
        "source": "saved_template",
        "exerciseId": "custom",
        "savedWorkoutId": "template-1001",
        "name": "下班后跳绳",
        "description": "自定义模板",
        "mode": "count_rate",
        "intensity": "moderate",
        "accentColor": "#7B61FF"
      }
    ]
  }
}
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `source` | `preset` 或 `saved_template` |
| `exerciseId` | 预设训练 ID；模板固定为 `custom` |
| `savedWorkoutId` | 只有模板项会有值 |
| `mode` | 对应训练详情页的输入模式 |
| `intensity` | 仅模板或自定义训练有意义 |

### 9.3 获取训练计划

`GET /api/v1/workout-plans?startDate=2026-04-01&endDate=2026-04-30`

说明：

- 返回当前月或当前可视范围内的所有计划。
- 前端会自行统计每一天的计划数，并渲染当日计划列表。

响应体：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "plans": [
      {
        "id": "plan-1001",
        "date": "2026-04-17",
        "title": "晚间跑步",
        "summary": "配速 + 路程 · 30 分钟 · 286 kcal",
        "notes": "下班后训练",
        "draftText": "running%7C%7Cduration%7Cmoderate%7C30%7C5%7C6%7C18%7C300%7C120%7C4%7C40%7C20",
        "savedWorkoutId": "",
        "accentColor": "#0F6BFF"
      }
    ]
  }
}
```

### 9.4 新增训练计划

`POST /api/v1/workout-plans`

请求体：

```json
{
  "date": "2026-04-17",
  "title": "晚间跑步",
  "notes": "下班后训练",
  "savedWorkoutId": "",
```

```json
  "exerciseId": "running",
  "name": "跑步",
  "calories": 286.4,
  "durationSeconds": 1800,
  "planId": "plan-1001",
  "configSnapshot": {
    "exerciseId": "running",
    "customName": "",
    "customMode": "duration",
    "customIntensity": "moderate",
    "durationMinutes": "30",
    "distanceKm": "5",
    "paceMinutes": "6",
    "speedKmH": "18",
```

- `configSnapshot` 强烈建议保存，后面做统计和复盘会很有用。

### 10.2 获取训练历史

`GET /api/v1/workout-records?startDate=2026-04-01&endDate=2026-04-17&pageNo=1&pageSize=20`

### 10.3 获取模板列表

`GET /api/v1/workout-templates`

### 10.4 新增模板

`POST /api/v1/workout-templates`

请求体：

```json
{
  "name": "下班后跳绳",
  "mode": "count_rate",
  "intensity": "moderate",
  "durationMinutes": "30",
  "distanceKm": "5",
  "paceMinutes": "6",
  "speedKmH": "18",
  "count": "500",
  "cadencePerMinute": "120",
  "sets": "4",
  "workSeconds": "40",
  "restSeconds": "20"
}
```

说明：

- 当前前端自定义模板的本质就是一份 `custom` 训练草稿。
- 建议后端给模板名加唯一约束：`userId + name`。

### 10.5 更新模板

`PUT /api/v1/workout-templates/{templateId}`

请求体与新增一致。

### 10.6 删除模板

`DELETE /api/v1/workout-templates/{templateId}`

## 11. 枚举值

### 11.1 `exerciseId`

| 值 | 含义 |

| `cycling` | 骑行 |
| `jump_rope` | 跳绳 |
| `yoga` | 瑜伽 |
| `strength` | 力量循环 |
| `custom` | 自定义运动 |

### 11.2 `customMode`

| 值 | 含义 |
| --- | --- |
| `duration` | 按时长 |
| `set_timer` | 按组计时 |
| `distance_pace` | 配速 + 路程 |
| `distance_speed` | 均速 + 路程 |
| `count_rate` | 次数 + 节奏 |

### 11.3 `customIntensity`

| 值 | 含义 |
| --- | --- |
| `light` | 轻松 |
| `moderate` | 中等 |
| `intense` | 高强 |

### 11.4 `mealType`

| 值 | 含义 |
| --- | --- |
| `breakfast` | 早餐 |
| `lunch` | 午餐 |
| `dinner` | 晚餐 |
| `snack` | 加餐 |

### 11.5 `gender`

当前前端直接使用中文值：

| 值 | 含义 |
| --- | --- |
| `男` | 男 |
| `女` | 女 |
| `未填写` | 未填写 |

说明：

- 如果后端想统一成 `male/female/unknown`，前端需要补映射。
- 首版联调建议直接兼容中文值。

### 11.6 `task`

| 值 | 含义 |
| --- | --- |
| `chat` | 普通问答 |
| `analyze_weight` | 体重趋势分析 |
| `generate_plan` | 生成训练计划草案 |

## 12. 当前前端校验规则

### 12.1 用户资料

| 字段 | 规则 |
| --- | --- |
| `nickname` | 可空 |
| `heightCm` | 可空，填了就参与 BMI |
| `weightKg` | 可空，填了就参与 BMI 和热量估算 |
| `age` | 可空 |
| `gender` | `男`、`女`、`未填写` |

### 12.2 体重记录

| 字段 | 规则 |
| --- | --- |
| `weight` | 必须是数字，范围 20 到 300 |

### 12.3 训练记录与模板

| 场景 | 规则 |
| --- | --- |
| `exerciseId = custom` | `customName` 必填 |
| `customMode = duration` | `durationMinutes > 0` |
| `customMode = distance_pace` | `distanceKm > 0` 且 `paceMinutes > 0` |
| `customMode = distance_speed` | `distanceKm > 0` 且 `speedKmH > 0` |
| `customMode = count_rate` | `count > 0` 且 `cadencePerMinute > 0` |
| `customMode = set_timer` | `sets > 0` 且 `workSeconds > 0` |
| 结束并记录 | `durationSeconds > 0` |

### 12.4 食物计划

建议后端同步做以下基础校验：

| 字段 | 规则 |
| --- | --- |
| `name` | 自定义食物名称必填 |
| `calories` | 大于 0 |
| `serving` | 建议必填，方便前端直接展示 |
| `mealType` | 必须在 `breakfast/lunch/dinner/snack` 内 |

## 13. 联调建议

### 13.1 第一阶段先打通这 5 条链路

1. 用户页读写资料。
2. 体重页按日期保存和查询体重列表。
3. 训练页先拉 `workout-types`，再拉 `workout-plans`。
4. 训练详情页保存训练记录、模板增删改查。
5. AI 助手页走一个统一 `chat` 接口，先返回 `reply + plans`。

### 13.2 第二阶段再补增强能力

1. 食物页正式接入页面和 Tab。
2. 训练计划后端同时保存 `draftText` 和结构化 `draft`。
3. AI 助手由后端自动拼接用户上下文，前端不再上传 `contextSnapshot`。
4. 增加饮食与训练联合分析能力，供 AI 返回更完整建议。

### 13.3 唯一键建议

| 模块 | 唯一键建议 |
| --- | --- |
| 用户资料 | `userId` |
| 体重记录 | `userId + date` |
| 饮食计划 | `id` |
| 训练计划 | `id` |
| 训练模板 | `userId + name` 或 `id` |
| 训练记录 | `id` |
